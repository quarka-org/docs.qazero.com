# `goal_N` — conversion / goal イベント

`goal_N` は単一のマテリアルではなく、**マテリアルのファミリー** です。各サイトは最大10個のゴールを定義でき、それぞれが `goal_1`, `goal_2`, …, `goal_10` という名前のマテリアルになります。たとえば `goal_3` の1行は「このサイトで誰かがゴール#3を発火した」を意味します。

**コンバージョン** についての質問には `goal_N` を使ってください: 何件ゴールが発火したか、どのページが連れてきたか、どのリファラーか、コンバージョンまでの経路はどうだったか。ユーザーが「コンバージョン」「サインアップ」「購入」「リード」「お問い合わせ」と言ったら、`goal_N` を考えてください。

## どの goal が何か?

`/guide` エンドポイントは `sites` 配列を返し、各サイトは `goals` リストを持ち、各ゴールは `id`, `name`, 人間が読める条件を含みます。これが「どのゴール番号が何を意味するか」を引くところです。**`goal_1` が "signup" だと絶対に推測しないでください** — サイトオーナーがそれぞれのサイトでどう設定したかに依存します。

典型的な `sites[0].goals` レスポンス:

```json
[
  { "id": 1, "name": "お問い合わせ送信",  "type": "gtype_page",
    "condition": { "url": "/contact/thanks", "match": "exact" } },
  { "id": 2, "name": "メルマガ登録",       "type": "gtype_click",
    "condition": { "page": "/", "selector": "#newsletter-submit" } },
  { "id": 3, "name": "無料トライアル開始", "type": "gtype_page",
    "condition": { "url": "/trial/start",   "match": "exact" } }
]
```

ユーザーが「今週何件サインアップ来た?」と訊いたら、このリストを見て、`name` が "メルマガ登録" のゴールを選び (または意図に合うもの)、対応する `goal_N` マテリアルをクエリします。

## サンプル行

`goal_3` ("無料トライアル開始") の手作りサンプル。ID カラムは `page_id`, `session_id`, `pv_id` のみ。

| pv_id | session_id | page_id | access_time         | url                  | referrer         | device  | browse_sec |
|------:|-----------:|--------:|---------------------|----------------------|------------------|---------|-----------:|
| 91030 | 48221      | 1305    | 2026-04-14 09:14:22 | `/trial/start`       | google / organic | desktop |         38 |
| 91033 | 48224      | 1305    | 2026-04-14 10:02:11 | `/trial/start`       | (ダイレクト)        | mobile  |         72 |
| 91041 | 48228      | 1305    | 2026-04-14 11:48:55 | `/trial/start`       | twitter.com      | mobile  |         24 |
| 91048 | 48230      | 1305    | 2026-04-14 13:22:07 | `/trial/start`       | google / ads     | desktop |         19 |

一目で読み取れる:

- すべての行で `page_id` (1305) と URL (`/trial/start`) が同じ。これは期待通り: `goal_3` はページ型ゴールなので、発火はすべて thanks/start ページの閲覧。
- 行の違いは「誰がどうやって来たか」: organic / ダイレクト / ソーシャル / ads のミックス。これがこのマテリアルを有用にしている理由 — `allpv` 的なカラムのどれでもコンバージョンを分解できる。

すべての `goal_N` マテリアルは `allpv` と同じ形を持ちますが、そのゴールを発火したページビューだけに絞り込み済みです。つまり source、device、referrer、engagement カラムを得るために **JOIN は不要** で、最初から各行に乗っています。

## よく使う JOIN キー

- `pv_id` → `click_event.pv_id` (コンバージョンページビューで何をクリックしたか)
- `session_id` → `allpv.session_id` (コンバージョン前にセッション内で見た他のページ)
- `page_id` → `gsc.page_id` (オーガニック検索から来たか)

## `goal_N` に向いた最初の質問

- 「今週何件コンバージョンしたか、source 別に?」
- 「コンバージョン前の browse_sec が長い = 質の高いコンバージョンを生むリファラーは?」
- 「デバイス種別ごとのコンバージョン率は?」
- 「2ステップ先で一番コンバージョンを生むランディングページは?」
